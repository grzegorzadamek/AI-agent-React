Backend API schema for the auth + dashboard flow
================================================

This project currently uses a mock auth flow. To connect a real backend, the frontend should call backend endpoints instead of the mock functions in src/lib/mockApi.ts.

Recommended security model
--------------------------
- Use OAuth 2.1 / Authorization Code + PKCE.
- Exchange the Google authorization code on the backend.
- Return short-lived access tokens and refresh tokens.
- Store tokens securely in HttpOnly, Secure, SameSite cookies.
- Validate access tokens on every protected request on the backend.
- Rotate refresh tokens and revoke them on logout.

Suggested backend endpoints
---------------------------

1) GET /api/auth/google/login
Purpose:
- Start the OAuth login flow.

Behavior:
- Redirect the browser to Google OAuth authorize endpoint.
- Include state and code_verifier generated by the backend/client.
- Return 302 redirect.

Example:
- GET /api/auth/google/login
- Response: 302 Location: https://accounts.google.com/o/oauth2/v2/auth?... 

2) GET /api/auth/google/callback
Purpose:
- Finish Google login after user returns from Google.

Query params:
- code
- state
- error (optional)

Behavior:
- Validate state and code_verifier.
- Exchange authorization code for user profile from Google.
- Create or update user record.
- Issue access token + refresh token.
- Set HttpOnly cookies or return JSON.

Success response:
- Status: 200 OK
- Body:
  {
    "authorized": true,
    "user": {
      "id": "google-user-id",
      "name": "Jan Kowalski",
      "email": "jan@example.com",
      "role": "Product Designer",
      "plan": "Pro",
      "avatar": "JK"
    },
    "expiresIn": 900
  }

Security notes:
- Validate nonce/state.
- Verify issuer, audience, and expiry of the Google ID token or access token.
- Never trust the frontend input alone.

3) POST /api/auth/refresh
Purpose:
- Rotate access token using refresh token.

Headers:
- Cookie: refresh_token=...

Body (optional if using cookies):
{
  "refreshToken": "..."
}

Success response:
- Status: 200 OK
- Body:
  {
    "accessToken": "new-access-token",
    "expiresIn": 900
  }

4) POST /api/auth/logout
Purpose:
- Invalidate current session and revoke refresh token.

Headers:
- Cookie: access_token=...; refresh_token=...

Success response:
- Status: 204 No Content

5) GET /api/me
Purpose:
- Return current authenticated user profile.

Headers:
- Authorization: Bearer <accessToken>
OR
- Cookie: access_token=...

Success response:
- Status: 200 OK
- Body:
  {
    "id": "google-user-id",
    "name": "Jan Kowalski",
    "email": "jan@example.com",
    "role": "Product Designer",
    "plan": "Pro",
    "avatar": "JK"
  }

6) GET /api/dashboard
Purpose:
- Return dashboard statistics for the authenticated user.

Headers:
- Authorization: Bearer <accessToken>
OR
- Cookie: access_token=...

Success response:
- Status: 200 OK
- Body:
  {
    "projects": 12,
    "tasks": 34,
    "notifications": 7,
    "completion": 84
  }

Error responses:
- 401 Unauthorized: missing or invalid access token
- 403 Forbidden: authenticated user lacks permission
- 401/403 depending on your policy when session is expired

Suggested frontend mapping to current mock flow
----------------------------------------------
- authenticateUser(...) -> POST /api/auth/google/callback or GET /api/auth/google/callback
- refreshAccessToken(...) -> POST /api/auth/refresh
- fetchDashboardData(...) -> GET /api/dashboard

What still needs to be done before production
--------------------------------------------
1. Replace the mock API with real HTTP requests.
2. Switch from sessionStorage/localStorage token storage to HttpOnly cookies.
3. Implement Authorization Code + PKCE for Google login.
4. Validate access tokens on the backend for every protected route.
5. Add refresh-token rotation and revocation on logout.
6. Add CSRF protection for cookie-based auth.
7. Add rate limiting, audit logging, and monitoring.
8. Use HTTPS in production.
9. Add role-based authorization checks for admin/user routes.
10. Add tests for happy path, expired token, forbidden user, and logout flow.

Recommended response formats
----------------------------
Use consistent JSON payloads:

Auth success:
{
  "success": true,
  "data": {
    "user": {...},
    "expiresIn": 900
  }
}

Error:
{
  "success": false,
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Invalid or expired session"
  }
}

Minimal implementation checklist
-------------------------------
- Backend route for Google OAuth start
- Backend route for Google OAuth callback
- Access token issuer + validator
- Refresh token store and rotation
- Protected dashboard route
- Logout route that clears cookies
- CORS configuration for the frontend origin
- Environment variables for Google client ID/secret and JWT secret
